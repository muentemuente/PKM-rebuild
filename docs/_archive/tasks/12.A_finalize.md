## AP3 — Vocab-Management + inkrementeller Workflow

Ziel (deine Vorgabe): neue Files in definierten Ordner legen → automatisiert durch alle Parts → Auswertung → category/tag vergeben (neu oder bestehend) → in Vault → Docs aktualisieren.

### 3a — Inbox-Ordner + inkrementeller Lauf
- **Inbox:** `data/00_inbox/` (neue Roh-`.md`). Definiert, dokumentiert, `.gitignore`-abgedeckt.
- **Inkrementeller Modus:** `python -m pipeline ingest` (oder `run --incremental`):
  - verarbeitet **nur** Files aus `00_inbox/` durch Phasen 1→8 (Routing wie Option B: passthrough/stage3/gedanken).
  - bestehenden Vault + bestehende Drafts **nicht** anrühren (Hash-Skip-Logik nutzen).
  - Output: neue Drafts in `03_drafts/`, plus **Auswertungs-Report** `02_pipeline_output/ingest_report.md`: pro neuem Doc → vorgeschlagene `category` + `tags`, Flag ob category ∈ bestehende 17 / Tag ∈ Vokabular oder NEU.
- Nach erfolgreicher Vault-Übernahme: verarbeitete Files `00_inbox/` → `01_corpus_input/` verschieben (werden Teil des Korpus). Manifest-reconcile.

### 3b — `scripts/manage_vocab.py` (Helper, idempotent)
- **add-category `<name>`:** legt neue category konsistent an allen Stellen an: `CATEGORY_TO_FOLDER` (+ nächste freie `NN_`-Nummer), `ALLOWED_CATEGORIES`, Vault-Ordner anlegen, `03_vault_standard.md` §4 + E5-Appendix ergänzen. Validierung: Slug-Regeln, keine Kollision.
- **add-tag `<tag>`:** ergänzt kontrolliertes Vokabular in `00_Meta/tag-system.md` mit Begründungs-Feld (Vault-Standard §7).
- **list / validate:** zeigt aktuelles Vokabular, prüft Drift (category 3 Stellen konsistent? Tags ∈ Vokabular?).
- Konsistenz-Test gegen `_pkm_common`/`pipeline.schemas`.

### 3c — Übernahme-Pfad
- Dokumentierter Ablauf in `docs/FUTURE_RUN.md` (umschreiben von „nach Projektabschluss" → „Standard-Workflow inkrementelle Verarbeitung"):
  1. Files → `00_inbox/`
  2. `pipeline ingest` → `ingest_report.md` lesen
  3. ⏸ **REVIEW (Mensch):** neue category/tag? → `manage_vocab.py add-*` ODER bestehende zuordnen (Frontmatter-Edit/`apply_category_mapping`)
  4. `build-vault` (nur neue Drafts ergänzen)
  5. `reports` regenerieren
  6. Docs-`updated:` ziehen

### Tests
- inkrementeller Lauf: 1–2 synthetische Inbox-Files → durch alle Phasen, bestehender Vault unverändert, `ingest_report.md` korrekt (neu-vs-bestehend-Flags).
- `manage_vocab.py`: add-category konsistent an 3 Stellen; add-tag im Vokabular; validate erkennt künstlichen Drift.
- Idempotenz: ingest 2× ohne neue Files = no-op.

**Probelauf (Dry):** 1 Dummy-`.md` in `00_inbox/` → `ingest --dry-run` → Report zeigt Vorschlag, schreibt nichts. Dummy danach entfernen.
Commit `feat(pipeline): incremental ingest + manage_vocab + inbox workflow`.
